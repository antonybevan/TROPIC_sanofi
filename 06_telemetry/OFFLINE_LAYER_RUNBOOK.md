# Offline Capability-Layer Runbook

The 20-stage pipeline (`06_telemetry/cibuild.py`, gated by `.github/workflows/ci.yml`)
builds and validates the core ADaM/SDTM/TFL/define/eCTD-`m5` deliverables and — as
post-stages 17–19 — the modern machine-readable exports (Dataset-JSON / ARS / USDM). A
remaining **additive capability layer** runs on demand beside it: the SDTMIG-3.4 uplift,
the DTD-valid eCTD sequence, the date-precision sensitivity analysis, and the CDISC CORE
conformance run.

**Wiring status.** The three modern exports — `export_datasetjson.py` (stage 17),
`build_ars.py` (18), `build_usdm.py` (19) — are **now manifest post-stages**, each gated by
its own exit code in the full pipeline; `build_usdm.py` (data-free) additionally runs as a
CI gate. They remain documented here for **standalone reproduction**. The other steps stay
**standalone — run on demand, not orchestrated by `cibuild.py` and not gated in CI** — for
two reasons:

1. **Data.** Most read patient-level XPT (`04_adam/*_prod.xpt`, `01_raw_source/real_sdtm/`,
   the `m5/` tree). This repository is **data-free** (those files are git-ignored), and CI
   has no real data, so these steps cannot execute there. CI runs only the data-free
   well-formedness gates (`validate_core_rules.py`, `validate_define.py`, the demo smoke
   test). The full engine/data runs are this local runbook.
2. **Secrets / heavyweight deps.** The CORE run needs Python 3.12, a free CDISC Library
   API key, network access, and a ~127 MB engine clone (`run_core_conformance.sh`).

This file is the single index of that layer so it does not rot or become unclear how to
regenerate. Detailed per-step records live in `11_ectd/RUN_RECORD.md`,
`06_telemetry/conformance/CORE_*RUN_RECORD.md`, and `REPRODUCIBILITY.md §7`.

## Prerequisites

- The CI pipeline has run and the real ADaM/SDTM data + `m5/` tree are present locally
  (these are the inputs the capability layer reads).
- **R** with `renv::restore()` (haven, dplyr) — for the SDTM uplift.
- **Python 3** with `lxml`, `pyyaml`, `pyreadstat`; `pip install usdm` for USDM; Python
  **3.12** + `CDISC_LIBRARY_API_KEY` for CORE (see `REPRODUCIBILITY.md §7`).

## Run order

Run top-to-bottom; later steps consume earlier outputs (the SDTM 3.4 uplift feeds the eCTD
`m5` SDTM copies and the SDTM Dataset-JSON; backbone precedes materialize).

| # | Script | Reads | Writes | Real data? | In pipeline? |
|---|---|---|---|---|---|
| 1 | `06_telemetry/uplift_sdtm_34.R` | `01_raw_source/real_sdtm/*.sas7bdat` (3.1.1, pristine) | `.core_run/sdtm34/*.xpt` + `m5/.../sdtm/datasets/*.xpt` (3.4) | yes | offline |
| 2 | `07_define_xml/uplift_define_34.py` | `07_define_xml/define_sdtm.xml` (+ embedded 3.4 column metadata) | `07_define_xml/define_sdtm.xml` (SDTMIG 3.4, CT 2026-03-27) | no (metadata-only) | offline |
| 3 | `06_telemetry/export_datasetjson.py` | `04_adam/*_prod.xpt`, SDTM `*.xpt` | `10_datasetjson/**/*.json` (Dataset-JSON v1.1) | yes | **stage 17** |
| 4 | `06_telemetry/build_ars.py` | MP-arm KM results | `12_ars/` (ARS v1.0 ReportingEvent + ARD) | yes | **stage 18** |

```bash
# data layer (1–2)
Rscript 06_telemetry/uplift_sdtm_34.R
python3 07_define_xml/uplift_define_34.py
# modern machine-readable exports (3–5)
python3 06_telemetry/export_datasetjson.py
python3 06_telemetry/build_ars.py
python3 06_telemetry/build_usdm.py
# eCTD sequence (6–7)
python3 06_telemetry/build_ectd_backbone.py
python3 06_telemetry/materialize_ectd.py
# sensitivity + conformance (8–9)
python3 06_telemetry/date_precision_sensitivity.py
export CDISC_LIBRARY_API_KEY=<free CDISC account key>
bash 06_telemetry/run_core_conformance.sh
```

## Determinism / what is committed

Each step is deterministic for fixed inputs and **additive** — none modifies the pristine
source or the CI-built deliverables. Committed artifacts are the data-free outputs and the
run records (backbone XML, STF, define, run records); patient-level outputs (XPT,
Dataset-JSON, CORE reports, materialized payload) are git-ignored and regenerated by the
steps above.
